class="section" id="hello-world-app"> <span id="createapp"></span><h1>Hello World App<a class="headerlink" href="#hello-world-app" title="Permalink to this headline">¶</a></h1> <p>Once you’ve created a developer account, you’re ready to create your first app.</p> <img alt="_images/sign_overview_2.png" src="_images/sign_overview_2.png" /> <div class="section" id="create-an-app-in-the-ui"> <h2>Create an app in the UI<a class="headerlink" href="#create-an-app-in-the-ui" title="Permalink to this headline">¶</a></h2> <p>To use Adobe Sign APIs, create an application.</p> <ol class="arabic simple"> <li><p><a class="reference external" href="https://secure.adobesign.com/public/login">Log in to Adobe Sign</a>.</p></li> <li><p>Select <strong>API</strong> from the top menu. If you are already an enterprise customer, you may not see the API link. In that case, choose <strong>Account</strong>.</p></li> <li><p>Select <strong>API Applications.</strong></p></li> </ol> <img alt="_images/sign_gstarted_1.png" src="_images/sign_gstarted_1.png" /> <ol class="arabic simple" start="4"> <li><p>Select the <strong>Create</strong> (+) icon and provide details about your app, including the intended use:</p></li> </ol> <blockquote> <div><ul class="simple"> <li><p><strong>CUSTOMER:</strong>: Apps for internal use and testing.</p></li> <li><p><strong>PARTNER:</strong>: Select this type if you’re developing an application for other users.</p></li> </ul> </div></blockquote> <div class="admonition note"> <p class="admonition-title">Note</p> <p>PARTNER applications will not have full access to other accounts until they have been certified. <a class="reference external" href="https://www.adobe.com/go/esign-dev-cert">Click here to get your application certified</a>.</p> </div> <img alt="_images/sign_gstarted_3.png" src="_images/sign_gstarted_3.png" /> </div> <div class="section" id="configure-oauth"> <span id="oauth"></span><h2>Configure OAuth<a class="headerlink" href="#configure-oauth" title="Permalink to this headline">¶</a></h2> <p>OAuth is widely used by developers to authorize API access. Adobe Sign’s implementation complies with the <a class="reference external" href="https://tools.ietf.org/html/rfc6749">OAuth 2.0 specification</a>. Adobe Sign uses the OAuth authentication protocol to authorize requests for any Sign API endpoint. Get the Application ID and Application Secret from the web UI of Adobe Sign.</p> <p>At a high level, the OAuth process requires the client application to request permissions from the end user before performing any actions on their behalf. This is accomplished by redirecting users to the Adobe Sign application where they will authenticate themselves and grant the requested permissions. Afterwards, the user is redirected back to the client web application.</p> <p>The OAuth process returns Access Tokens which can be used to make <a class="reference external" href="https://www.adobe.com/go/acrobatsignapireference">REST API calls</a>. Once you have created your application, click on the “Configure OAuth” link to configure your OAuth integration. This page contains the following information:</p> <ul class="simple"> <li><p>Your “Client Id” and “Client Secret”, which you will need during the token exchange.</p></li> <li><p>You will need to specify the “Redirect URI”, which is the page on your website we’ll return users after the OAuth flow.</p></li> <li><p>You will need to select which scopes you want to enable for your application.</p></li> </ul> <div class="section" id="get-the-app-id-and-secret"> <h3>Get the app ID and secret<a class="headerlink" href="#get-the-app-id-and-secret" title="Permalink to this headline">¶</a></h3> <ol class="arabic simple"> <li><p>Select <strong>API Applications</strong> to view the list of apps you have created; then select your app to view its action menu.</p></li> </ol> <img alt="_images/apiapps.png" src="_images/apiapps.png" /> <ol class="arabic simple" start="2"> <li><p>Select <strong>View/Edit</strong> to get the application ID and secret.</p></li> <li><p>Save the app’s application ID and secret. You’ll use this information to issue access tokens in the Adobe Sign API.</p></li> </ol> <img alt="_images/appsecret.png" src="_images/appsecret.png" /> </div> <div class="section" id="set-the-app-scopes"> <h3>Set the app scopes<a class="headerlink" href="#set-the-app-scopes" title="Permalink to this headline">¶</a></h3> <p>Scopes describe what resources and actions your application will access. Common scenarios include:</p> <ul> <li><p>As a developer, you will need to enable the appropriate scopes for your application on the admin page.</p> <blockquote> <div><ul class="simple"> <li><p>This should be the minimal set of scopes necessary for your application to function.</p></li> <li><p>If you apply for Certification, Adobe will review the enabled scopes to confirm that they match the intended application use.</p></li> </ul> </div></blockquote> </li> <li><p>Your application will make an OAuth authorization request with a scope parameter which indicates what permissions are being requested at that time.</p> <blockquote> <div><ul class="simple"> <li><p>The requested scopes must be a subset of the scopes that are enabled for the application.</p></li> <li><p>The requested scopes must be appropriate for the action the user is attempting to perform.</p></li> <li><p>The user will be asked to authorize the requested permissions for your application.</p></li> </ul> </div></blockquote> </li> <li><p>Assuming the OAuth tokens were successfully received, your application will make further API calls using the Access Token.</p> <blockquote> <div><ul class="simple"> <li><p>The calls that your application makes must be permitted by the scopes associated with the Access Token that’s being used.</p></li> <li><p>For example, in order to call GET /agreements, the agreement_read scope must have been requested and authorized.</p></li> </ul> </div></blockquote> </li> </ul> <ol class="arabic simple"> <li><p>Select <strong>Configure OAuth for Application</strong> to enable scopes.</p></li> </ol> <div class="admonition tip"> <p class="admonition-title">Tip</p> <p>The requested permissions on this screen sets the max allowable scopes, but every token is not going to exercise them all. Different tokens for various users in specific scenarios may use one or more of the the scopes (but not all).</p> </div> <img alt="_images/scopes.png" src="_images/scopes.png" /> <p>Remember to limit the scopes that you intend to request through OAuth. The scope denotes the extent of permission that your users will have on your account. For example, <code class="docutils literal notranslate"><span class="pre">agreement_send:account</span> <span class="pre">user_login:self</span></code> would allow the application to send on behalf of any user in the account and also login on behalf of the user that authorized the request.</p> <p>The following modifiers are available:</p> <blockquote> <div><table class="docutils align-default"> <colgroup> <col style="width: 13%" /> <col style="width: 87%" /> </colgroup> <thead> <tr class="row-odd"><th class="head"><p><strong>Modifier</strong></p></th> <th class="head"><p><strong>Description</strong></p></th> </tr> </thead> <tbody> <tr class="row-even"><td><p><code class="docutils literal notranslate"><span class="pre">self</span></code></p></td> <td><p>Perform the specified action on behalf of the authorizing user. This is the default: for example, “agreement_send:self” is the same scope as “agreement_send”</p></td> </tr> <tr class="row-odd"><td><p><code class="docutils literal notranslate"><span class="pre">group</span></code></p></td> <td><p>Perform the specified action on behalf of any user in the same group as the authorizing user. The authorizing user must be a group admin in order to grant this scope, and must have the Business or Enterprise edition of Adobe Sign.</p></td> </tr> <tr class="row-even"><td><p><code class="docutils literal notranslate"><span class="pre">account</span></code></p></td> <td><p>Perform the specified action on behalf of any user in the same account as the authorizing user. The authorizing user must be an account admin in order to grant this scope, and must have the Business or Enterprise edition of Adobe Sign.</p></td> </tr> </tbody> </table> </div></blockquote> <p>Now you can invoke the Sign APIs from your client-side apps.</p> </div> </div> <div class="section" id="oauth-token-exchange"> <span id="token"></span><h2>OAuth token exchange<a class="headerlink" href="#oauth-token-exchange" title="Permalink to this headline">¶</a></h2> <div class="section" id="authorization-request"> <h3>Authorization Request<a class="headerlink" href="#authorization-request" title="Permalink to this headline">¶</a></h3> <p>The OAuth process starts with the client directing the user’s browser to make a request to the /public/oauth/v2 endpoint, with the following query string parameters:</p> <table class="colwidths-given docutils align-default"> <colgroup> <col style="width: 10%" /> <col style="width: 20%" /> <col style="width: 5%" /> <col style="width: 65%" /> </colgroup> <thead> <tr class="row-odd"><th class="head"><p>Parameter</p></th> <th class="head"><p>Value</p></th> <th class="head"><p>Required?</p></th> <th class="head"><p>Description</p></th> </tr> </thead> <tbody> <tr class="row-even"><td><p>response_type</p></td> <td><p>code</p></td> <td><p>yes</p></td> <td><p>The value must always be <em>code</em></p></td> </tr> <tr class="row-odd"><td><p>client_id</p></td> <td><p>Your ID obtained from OAuth configuration page.</p></td> <td><p>yes</p></td> <td><p>Identifies the application</p></td> </tr> <tr class="row-even"><td><p>redirect_uri</p></td> <td><p>A secure, absolute URI, for e.g. <a class="reference external" href="https://example.com">https://example.com</a>.</p></td> <td><p>yes</p></td> <td><p>Redirects users here at the end of the authorization process. The value must belong to the set of values specified on the OAuth Configuration page.</p></td> </tr> <tr class="row-odd"><td><p>scope</p></td> <td><p>A space delimited set of permissions, specified during the OAuth configuration setup on the Configure OAuth page.</p></td> <td><p>yes</p></td> <td><p>The permissions that the user will be asked to approve</p></td> </tr> <tr class="row-even"><td><p>state</p></td> <td><p>Any string</p></td> <td><p>no</p></td> <td><p>This value will be returned to the client as a parameter at the end of the authorization process. While not required, use of the <code class="docutils literal notranslate"><span class="pre">state</span></code> parameter is highly recommended to protect against CSRF <a class="reference external" href="https://tools.ietf.org/html/rfc6749#section-10.12">as described in the RFC</a>.</p></td> </tr> </tbody> </table> <p>To invoke the Sign APIs, your application must first get an OAuth access token on behalf of an Adobe Sign user. This process uses HTTP requests to obtain the token. Your url will look like the following (with your client ID, etc.):</p> <div class="highlight-default notranslate"><div class="highlight"><pre><span></span>https://secure.adobesign.com/public/oauth/v2? response_type=code& client_id=CBAThIsIsNoTaReAlmPBvPF& scope=user_login:self+agreement_send:account& redirect_uri=https://myserver.com </pre></div> </div> <p>Next, select <strong>Allow Access</strong> to receive the authorization code:</p> <img alt="_images/sign_gstarted_6.png" src="_images/sign_gstarted_6.png" /> <p>Once you select Allow Access, the page gets redirected to the following URL:</p> <div class="highlight-default notranslate"><div class="highlight"><pre><span></span>https://myserver.com/? code=CBNCKBAThIsIsNoTaReAlcs_sL4K32wCzs4N </pre></div> </div> <p>Note the code <code class="docutils literal notranslate"><span class="pre">CBNCKBAThIsIsNoTaReAlcs_sL4K32wCzs4N</span></code> in the request. This is the authorization code that you will use to get the OAuth access token.</p> <p>You can send the authorization code along with the client ID and client secret to the Sign service to get the token. Use the same <code class="docutils literal notranslate"><span class="pre">api_access_point</span></code> that you got from the previous response.</p> <div class="highlight-default notranslate"><div class="highlight"><pre><span></span>POST https://api.adobesign.com/oauth/v2/token? grant_type=authorization_code& code=CBNCKBAThIsIsNoTaReAlcs_sL4K32wCzs4N& client_id=CBAThIsIsNoTaReAlmPBvPF& client_secret=319UThIsIsNoTaReAl2-4OxkVo9ycU& redirect_uri=https://myserver.com HTTP/1.1 Content-Type: application/x-www-form-urlencoded </pre></div> </div> <p>You will get the following JSON body containing the access token and the refresh token:</p> <div class="highlight-default notranslate"><div class="highlight"><pre><span></span><span class="p">{</span> <span class="s2">"access_token"</span><span class="p">:</span><span class="s2">"3AAABLblThIsIsNoTaReAlToKeNPr6Cv8KcZ9p7E93k2Tf"</span><span class="p">,</span> <span class="s2">"refresh_token"</span><span class="p">:</span><span class="s2">"3AAABLblThIsIsNoTaReAlToKeNWsLa2ZBVpD0uc*"</span><span class="p">,</span> <span class="s2">"token_type"</span><span class="p">:</span><span class="s2">"Bearer"</span><span class="p">,</span> <span class="s2">"expires_in"</span><span class="p">:</span><span class="mi">3600</span><span class="p">,</span> <span class="s2">"api_access_point "</span><span class="p">:</span><span class="s2">"https://api.na1.adobesign.com/"</span><span class="p">,</span> <span class="s2">"web_access_point"</span><span class="p">:</span><span class="s2">" https://secure.na1.adobesign.com/"</span> <span class="p">}</span> </pre></div> </div> <p>Note the following tokens in the response body:</p> <ul class="simple"> <li><p><strong>Access Token:</strong>: Use this token to access any Adobe Sign API endpoint.</p></li> <li><p><strong>Refresh Token:</strong> If your access token expires, use the refresh token to request for a new access token.
